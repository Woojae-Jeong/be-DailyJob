# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습
  - GET 요청의 특징
  - GET 요청에서의 쿼리스트링 전달 형태
  - HttpRequestParser에서의 쿼리스트링 파싱 로직 구현
  - 동적 요청 handler 설계
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 동적 요청 handler 구현

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - HTTP 파라미터를 저장하는 RequestParams의 저장 구조를 가변 제네릭 방식 대신, 일관된 Map<String, List<String>> 단일 리스트 형태로 고정했다
- 그렇게 판단한 이유:
  - 컴파일러가 타입 불일치를 잡지 못하는 제네릭 방식을 쓰면, 사용하기는 편할지언정 사용하는 측에서 해당 파라미터가 배열형태인지, 단수형태인지를 명확히 알아야 한다. 
  - 만약 제대로 알지 못한다면 런타임 중에 에러가 발생한다는 것을 알 수 있기에 택하지 않았다
  - 지금의 방식의 경우에는 List 형태로 일관되게 value를 관리하다보니 파라미터를 추가하는 로직이 간단해지고, 사용자가 해당 파라미터가 단수형태인지, 배열형태인지 알 필요가 없다 
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 파라미터를 추가하는 add()메소드의 동작이 간단해졌다는 점과 유지보수성 측면에서 해당 파라미터가 단수형태인지 복수형태인지에 대해서 반드시 알아야 하는 번거로움이 없어졌다

### 3. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지
  - RFC 문서를 읽어보니 쿼리스트링에 대한 규칙은 생각보다 널널하였다.
  - 따라서 다양한 요청 상황을 고려해서 쿼리스트링 파싱에 대한 내 서버만의 규칙을 세운 것이 오늘 잘한 행동이다
- 그 접근이 도움이 된 이유:
  - 항상 정답의 것을 찾아가다가 나만의 것을 설계해야되는 상황에 처음에는 당황하였고, 고려하지 못한 부분이 있지 않을까 걱정이 되었지만 오히려 올바르지 않은 요청의 상황을 다양하게 생각해보면서 시스템 설계에서의 예외 상황을 생각해내는 능력을 기를 수 있었다
- 다음에도 반복하고 싶은 점:
  - 예외 상황을 생각해내는 과정에서 나 혼자서 한 것이아닌 AI와 함께 생각해내었다. 다음에도 AI와 함께 예외 상황을 생각해내고 내 서버에서는 어떻게 처리할지 자기주도적으로 설계를 해 나가는 것을 반복하고 싶다

### 4. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점: 
  - 동적 handler를 설계하는 과정에서 제네릭을 어떤 상황에서 써야하는지에 대한 나만의 철학이 없다는 것을 발견하였다
- 그렇게 된 원인:
  - 제네릭을 이용하면 하나의 타입에 한정되지 않고 다양한 타입을 받을 수 있다는 것에 매몰되어 지금까지 나는 다양한 타입을 받을 수 있다면 제네릭을 많이 선택해왔다
  - 그래서 handler의 반환타입도 다양하다보니 제네릭으로 설계를 하였는데, 이게 오히려 동적 핸들러를 설계하는 과정에서 독이 되었다. 왜냐하면 Dispatcher쪽에서는 제네릭으로 타입을 받는다고 해서 특별하게 사용하는 로직이 없다보니 handler 인터페이스를 제네릭으로 가져가는게 맞나? 라는 생각이 들었다
  - 이런 상황에 도달하니 뇌정지가 왔고, 제네릭을 사용해야 하는 나만의 철학이 정립되지 않음을 깨달을 수 있었다
- 다음에는 다르게 해볼 구체적인 행동:
  - 아직 설계를 다 끝마치지 못했는데, 그 이전에 제네릭에 대한 철학과 주요 사용처에 대해서 찾아보고 나만의 제네릭에 대한 철학을 정립하려고 한다
