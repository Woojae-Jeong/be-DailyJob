# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - MIME Type 구조 공부
  - HttpResponse 클래스 내부에 HttpHeaders 추가
  - dot dot slash 공격 방어 로직 구현
  - ContentType enum에 이미지 관련 상수 추가
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 예외시스템 구축

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - HttpResponse 클래스의 멤버 변수로 HttpHeaders를 추가하고 ContentType을 삭제하는 구조로 리팩터링 한 것
- 그렇게 판단한 이유:
  - 기존 구조에서는 헤더가 추가될 때마다 enum 값으로 멤버변수로 추가할 구조로 잡았었기에 헤더 중에서 ContentType만을 가지고 있었다. 
  - 하지만 이러한 구조는 헤더가 추가될 때마다 enum값을 정의해줘야 하며, 그에 따라 HttpResponse의 수정이 발생하는 개방-폐쇄 원칙에 어긋나는 구조를 가지고 있었따
  - 그렇기에 헤더에 값을 추가하고, 삭제를 하더라도 코드 수정이 발생하지 않으면서 key : value 형태로 저장이 가능한 HttpHeaders 클래스를 vo 객체로 가지는 것이 개방-폐쇄 원칙을 만족한다고 판단하였다
- 고려했지만 선택하지 않은 대안:
  - 기존과 같이 중요한 헤더들에 대해서만 별도로 vo 클래스로 뽑아내려고 하였지만, 이는 새로운 클래스를 매번 만들어줘야 하며 개방-폐쇄 원칙에 어긋나기에 채택하지 않았다.
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 추가적인 헤더에 대해서도 문제가 없이 추가가 가능하며, 사용자가 쿠키를 추가하는 경우에도 key, value 형태로 간편하게 추가가 가능하다
  - 헤더가 추가가되더라도 HttpResponse의 구조에는 변함이 없다

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상: 
  - dot dot slash 방어 로직을 요청으로 들어온 uri 내부에 `../` 가 있는 구조로 설계를 하였다
  - 하지만 인코딩이 된 uri로 공격을 하게되면 뚫리는 구조임을 알게 되었다
- 처음 추측한 원인:
  - 현재 서버는 인코딩된 요청에 대해서 별다르게 디코딩하는 로직이 없기에 뚫리는 것이 당연했다.
- 원인을 확인하기 위해 시도한 방법:
  - 디코딩을 하고자 생각을 하였지만, 디코딩은 정적 요청 핸들러의 책임이 아니라고 판단하였다. 별도의 디코딩 기능을 서버에 추가하지 않고 dot dot slash 공격을 막을 수 있는 방법은 없는지 고민하였다.
- 실제 원인:
  - 설계에 대해서 고민을 하던 중, AI를 통해 보안 방법 중 하나로 금지하는 패턴을 막는 BlackList 방법도 있지만, 최종 결과물에 대해서 통과가 되는지만 검사하는 WhiteList 방법을 알게 되었다
- 해결 방법:
  - dot dot slash 공격을 WhiteList 방식으로 막는 것이 내가 놓치는 패턴들 없이 공격을 막아낼 수 있는 방법이라 판단하였다. 더불어 패턴들에 대한 관리가 필요 없으니 불필요하게 공격 패턴이 추가가 되더라도 코드의 수정은 발생하지 않는 유지보수에 좋은 구조였다
  - 따라서, BaseUrl에 요청으로 들어온 uri를 기준으로 최종 경로를 생성하고, 최종 경로가 BaseUrl로 시작 하는지 판단하는 간단한 로직으로 dot dot slash 공격을 방어하였다
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 보안과 관련된 문제를 해결해야 된다면, blackList 방법과 whitelist 방법을 둘 다 고려해서 각 방법마다의 트레이드 오프를 고려해 선택을 하고자 한다

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 각 클래스마다 예외적인 상황에 대해서 현재는 예외를 던지는 구조로 되어있다. 하지만 예외를 던져서 별도로 호출자쪽에서 catch 하는 구조로 예외를 처리하는 구조로 설계를 하지 않았다
  - 처음에는 호출자 쪽에서 예외를 모두 처리해야 된다고 생각을 하였지만, 이러한 구조로 가게되면, try-catch로 인해 코드의 가독성이 떨어질 것이라고 판단하였다. 
  - 더불어 예외를 받아서 처리를 하면 다행이지만, 예외를 받아서 별다른 로직 없이 또 다른 예외를 던지는 구조로 가게 된다면, 불필요한 코드가 삽입되는 것이라 판단하였다
  - 그렇기에 예외 처리가 발생하는 쪽에서는 예외를 던지기만 하고, 꼭 필요한 것이 아닌이상 최상위의 예외 시스템에서 예외를 처리하는 중앙 집중적인 예외 시스템을 설계하였다
- 그 접근이 도움이 된 이유:
  - 예외 처리를 호출자쪽에서 하는지, 아니면 별도의 책임을 가지는 클래스에서 중앙 집중형으로 처리를 하는 지에 따라서 코드의 가독성이 차이나는 것을 보고 코드 작성 과정에서의 시스템 설계가 중요함을 피부로 느낄 수 있었다
  - 또한 중앙 집중적인 구조가 책임을 분리하는 객체지향설계에 위배되지 않을까 염려가 되었지만, `예외 처리만을 담당한다` 라는 책임 하나만을 지게함으로써 중앙집중적인 구조가 객체지향설계에서도 올바른 선택일 수도 있음을 배웠다
- 다음에도 반복하고 싶은 점:
  - 책임을 분리할 때 각 클래스들의 공통된 책임들은 별도로 책임을 지는 클래스로 뽑아내는 구조도 설계시에 고민할 것이다 (산탄총 수술을 고려하자)
