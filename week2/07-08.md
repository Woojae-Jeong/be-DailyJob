# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
  - HttpRequest 파라미터, 경로 변수 파싱 기능 구현
  - 유저 회원가입 handler 구현
  - 요청을 라우팅 해주는 Router 구현
  - requestTarget을 path, 쿼리파라미터로 파싱하도록 HttpRequest 파싱 로직 수정
  - 제네릭, CQS 패턴에 대한 공부
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
  - 없음

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - handler를 찾는 과정에서 경로 변수를 추출하게 되는데, 이는 HttpRequest가 만들어진 뒤에 이루어진다.
  - 그래서 요청이 오고 Dispatcher에게 전달이 되기 전 HttpRequest를 만드는 로직에서 경로 변수를 뽑아내는게 객체지향적으로 좋은 설계로 처음에 판단하였다
  - 하지만, HttpRequest를 만든 뒤에 핸들러를 찾는 과정에서도 경로 변수를 추출하는 로직과 거의 동일한 로직을 사용하고 있어서, 코드 중복을 줄이면서 관심사 분리가 잘 이루어진 코드가 무엇인지 고민하게 되었다
  - 그 결과 HttpRequest를 만들 때에는 경로 변수를 추출하지 않고, 이후 Dispatcher가 Router를 통해서 라우팅을 하는 과정에서 handler 찾기는 물론 경로 변수 추출이 같이 이루어지게끔 하였다
- 그렇게 판단한 이유:
  - 경로 변수의 경우 route() 메소드를 거쳐야지만 uri 에서 경로변수라는 의미있는 정보를 뽑아낼 수 있다. 메소드를 실행하는 과정에서 추출된 정보 중 원래 이 메서드/책임이 원래 답해야 하는 질문에 그 정보가 포함이 된다면 반환을 해도 무방하다.
  - 따라서 반환되는 값이 책임에 위배되지 않게 클래스간 책임을 재분리하고, 메소드명도 메소드에서 추출된 정보가 반환되는 것이 이상하지 않게끔 메소드명을 잘 짓는 게 객체지향적인 코드임을 알게되었다
  - 따라서 기존에는 findHandler()라는 메소드명에서 match()라고 다시 짓고, 책임을 '라우트를 매칭시키고, 템플릿에 매칭됐을 때 매칭된 값을 반환한다'라고 다시 지으면서 핸들러를 찾고, 그 과정에서 매칭된 결과를 반환하는 것으로 객체지향적인 코드로 변경하였다
- 고려했지만 선택하지 않은 대안:
  - HttpRequest를 파싱할 때 같이 경로 변수를 먼저 파싱하고, 중복 코드를 감수하더라도 핸들러 매핑을 이후 따로 한번 더 호출하는 것을 고려하였지만 책임을 재정의하고 CQS 패턴에 위배되지 않게 match()메소드를 설계함으로써 해결하였다
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - 경로 변수 추출 로직이 한 곳에만 존재하게 되어, 코드를 중복으로 들고 있지 않게 되었다. 이후 수정이 일어난다고 한다면 이 부분만 수정을 하면 되는 유지보수성이 좋아졌다
  - 새로운 라우트(경로 변수가 있든 없든)가 추가돼도 Router의 등록 테이블에 한 줄 추가하는 것 외에 다른 코드를 건드릴 필요가 없어, 라우트 추가에 대해서 다른 코드에 변경이 발생하지 않는다

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 하나의 handler에서 여러 API(회원가입, 로그인 등)를 함께 처리하도록 설계하려 했는데, `Handler<T>` 인터페이스의 `handle()`이 API마다 다른 타입(`Void`, `SessionInfo` 등)을 리턴해야 해서 하나의 구체적인 T로 지정할 수 없었다. 결국 반환 타입을 와일드카드(`HttpResponse<?>`)로 열어두어야 하는 상황에 부딪혔다
- 처음 추측한 원인:
  - 제네릭을 인터페이스/클래스 레벨이 아니라 `handle()` 메소드 자체에 `<T>`를 붙이는 "메소드 레벨 제네릭"으로 내리면, 각 API마다 알맞은 타입을 자유롭게 리턴할 수 있을 거라고 생각했다.
- 원인을 확인하기 위해 시도한 방법:
  - 메소드 레벨 제네릭으로 바꿔서 작성을 해봤지만, 구체 타입을 캐스팅 없이 반환하려 하자 컴파일 에러가 발생했고, 언체크를 하기에는 제네릭의 `컴파일 단계에 타입을 강제해 타입 안정성을 가진다` 라는 장점을 챙길 수 없었다
- 실제 원인:
  - 메소드 레벨 제네릭의 타입 파라미터는 "구현부가 정하는 게 아니라 호출하는 쪽이 정하는 값"이라서, 하나의 handler 구현체가 특정 타입 하나로 마음대로 고정할 수 있는 구조가 아니었다. 근본적인 문제는 제네릭 문법 선택이 아니라, **"하나의 handler 인스턴스가 여러 개의 서로 다른 반환 타입을 동시에 책임지려 한 설계 자체"**였다.
- 해결 방법:
  - "하나의 handler는 하나의 API만 담당한다"로 설계를 바꿔서, 각 handler 클래스가 클래스 레벨의 제네릭을 구현하도록 했다. 
  - 이러면 클래스(인터페이스) 레벨 제네릭만으로 컴파일 타임 타입 안전성이 그대로 보장된다. 대신 API를 분기하는 책임은 Dispatcher/Router가 갖도록 재배치해서, 제네릭 문제와 단일 책임 원칙 위반을 동시에 해결했다.
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 제네릭과 다형성에 대해서 이번 계기로 어떤 상황에 쓰는 게 적합한지에 대해서 알게 되었다. 사실 지금과 같은 상황에서는 반환 타입에 대한 안정성을 챙길 수 있는 것이지, "제네릭이 가지는 여러 타입에 대해 동일한 코드를 적용시킬 수 있다" 라는 장점은 적용되지 못하고 있다
  - 그래서 다음번에는 제네릭과 다형성 중 어떤 것이 적합한지를 따져서 적용해보고자 한다

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - "핸들러를 찾는다" 에서 "Route를 매칭시키고, 그 매칭된 값을 받는다"로 Router의 책임을 재설계 한 것
- 그 접근이 도움이 된 이유:
  - CQS 패턴에 대해서 알게됌으로써 메소드의 동작이 "객체의 상태를 변경" or "결과 값을 반환" 둘 중 하나만 해야된다는 것을 알게되었다
  - 이러한 패턴을 알게되자 Router의 책임을 "핸들러를 찾는다"가 아닌 "Route를 매칭시키고, 핸들러와 그 매칭된 값을 반환한다"로 책임을 재설계 할 수 있었다
  - 책임을 단지 줄이려고 하는 게 아닌 확장을 시킴으로써 중복 코드를 줄일 수 있다는 것을 배웠고, 책임 분리는 물론이고 애초에 책임 설계를 잘해야 됌을 배울 수 있었다
- 다음에도 반복하고 싶은 점:
  - 다음번에도 이 클래스가 가지는 책임과 그 책임이 메소드를 통해서 구현이 될 때, 어떤 응답을 해야되는지에 대해서 고려한 후에 해당 클래스/메소드의 책임을 설계할 것이다

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 책임 분리 측면에서 경험이 부족했던 점이 아쉬웠다
- 그렇게 된 원인:
  - 단순하게 책임 분리라는 말에 꽂혀서 지금까지 책임을 작게 가져가는 것에만 생각했다
  - 하지만 책임은 결국 확대해서 가져갈 수 있는 것이고, 책임을 잘 분리한다는 건 "변하는 이유가 같은 것끼리 묶고, 변하는 이유가 다른 것은 분리하는 것"이란 걸 배울 수 있었다
- 다음에는 다르게 해볼 구체적인 행동:
  - 한 클래스/메소드의 책임을 설계할 때 과연 해당 메소드의 기능들이 변하는 이유가 같은 지를 따져보고 그렇지 않다면 책임을 분리해보도록 할 것이다
