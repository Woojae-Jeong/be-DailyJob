### 1. 오늘 완료한 것

    - Future의 한계점, 이를 극복하기 위해 나온 CompletableFuture의 개념 및 작업 종류 4가지

    - Fork/Join vs 일반 멀티쓰레드
    - 자바 Thread per Request 방식의 문제점 및 자바 가상 쓰레드 개념
      - Virtual Thread의 장단점
      - Virtual Thread의 구조
      - Virtual Thread의 Continuation

    - Concurrent 패키지의 주요 클래스

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
  - 오늘은 학습만 진행하였기에 가장 의미있었던 학습 과정에서의 판단은 자바 멀티쓰레드에 직접적으로 관련은 없지만 많은 동시성 클래스에서 내부적으로 이용하는 ForkJoinPool의 내부 동작에 대해 이해한점이다
- 그렇게 판단한 이유:
  - ForkJoinPool을 우리가 직접 서비스를 구현하면서 사용할 일은 없지만, Virtual Thread나 Stream API와 같이 여러 API에서 내부적으로 사용한다고 한다. 그렇기에 ForkJoinPool의 Fork, Join, WorkStealing에 대해서 이해함으로써 Virtual Thread의 스케줄러의 동작과 WorkStealing을 활용한 병렬 처리 로직을 잘 이해할 수 있었다
- 고려했지만 선택하지 않은 대안:
  - Virtual Thread를 직접 다루는 코드나 구체적인 성능 향상도를 조사해볼까 생각하였지만, 시간 관계상 알아보진 못했다. 
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
  - Java 21의 핵심이라고 할 수 있는 가상 쓰레드의 동작 과정에 대해서 잘 이해할 수 있었고, 이를 통해 전반적인 자바의 멀티스레드 개발방식에 대해서 감을 잡을 수 있었다

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
  - 쓰레드에 대해 공부하면서 동기화, 동기/비동기, 블로킹/논블로킹 단어가 나올 때마다 정확한 단어 정의를 모른다는 것을 인지했고, 이에 대해서 이해하는 과정에서 동기/비동기와 블로킹/논블로킹이 무슨 차이인지 이해하는데 어려움을 겪었다
- 처음 추측한 원인:
  - 기존에 동기/비동기를 메인 쓰레드에서 계속 하냐, 별도의 쓰레드로 진행되냐로, 블로킹/논블로킹을 메인쪽에서 중단이 되냐 안되냐로 이해하고 있던 개념을 결국 둘다 말이 다를 뿐 비슷한 개념이라고 생각했기 때문이다
- 원인을 확인하기 위해 시도한 방법:
  - [기술블로그](https://inpa.tistory.com/entry/%F0%9F%91%A9%E2%80%8D%F0%9F%92%BB-%EB%8F%99%EA%B8%B0%EB%B9%84%EB%8F%99%EA%B8%B0-%EB%B8%94%EB%A1%9C%ED%82%B9%EB%85%BC%EB%B8%94%EB%A1%9C%ED%82%B9-%EA%B0%9C%EB%85%90-%EC%A0%95%EB%A6%AC) 를 읽어보며 미묘한 차이가 무엇인지, 정확한 정의가 무엇인지 이해하려고 하였다
- 실제 원인:
  - 동기/비동기는 자신이 요청한 작업에 대해서 결과를 직접 기다리냐, 나중에 받느냐 즉, 작업 결과를 받는 방식의 차이다. 그리고 블로킹 논블로킹은 알고있던 대로 현재 작업이 블록되냐 안되냐이다. 
- 해결 방법:
  - 둘 사이에 어떤 차이점이 있는지 이해하고 동기/비동기 x 블로킹/논블로킹 조합으로 탄생하는 4가지 방식에 따른 동작 과정에 대해서 이해하니 차이를 이해할 수 있었다
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
  - 개념에 대해서 명확한 정의를 알고 있는지 체크를 해야한다. 그리고 이상하다는 느낌이 든다면 내가 알고 있는게 맞는지 한번 더 체크를 하자

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
  - 동기/비동기, 블로킹/논블로킹에 대해서 내가 알고있는것이 정확한지, 동기와 동기화는 무슨 차이가 있는건지 개념에 대한 더블 체킹을 한 행동
- 그 접근이 도움이 된 이유:
  - 내가 동기/비동기, 블로킹/논블로킹 중에서 어떤 점을 헷갈려하는지 메타인지를 할 수 있었다. 그리고 비슷한 개념이라도 어떤 차이가 있어서 나오게 된건지 이해해야 함을 배울 수 있었다
- 다음에도 반복하고 싶은 점:
  - 기술 블로그나 서버에 대해 학습할 때 내가 정말 말로 설명할 수 있을만큼 잘 이해하고 있는지 메타인지를 계속 시도하고자 한다

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
  - 동기/비동기, 블로킹/논블로킹에 대해서 약간 까먹은 부분이 있었지만 내가 알고 있는 것이 아마도 맞을거라는 추측을 가지고 자바 멀티 쓰레드 모델을 이해하려고 한 행동이 비효율적이였다
- 그렇게 된 원인:
  - 다시 찾아보기 귀찮다는 이유로 쓰레드 모델을 학습할 때 의미를 지레짐작 했었다. 이로 인해 쓰레드 모델을 다 학습한 이후에 두 개념에 대해서 찾아보는 이상한 행동을 하게 된 것이다
- 다음에는 다르게 해볼 구체적인 행동:
  - 귀찮다는 이유로 개념에 대해서 찾아보는 것을 미루는 행동을 하지 말고, 즉각적으로 찾아보고자 한다. 시간이 부족하더라도 결국 처음에 고생하는 것이 나중에 시간을 줄이는 행위다
