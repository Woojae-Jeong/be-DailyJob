# be-DailyJob

> 이 회고는 작업량을 보고하기 위한 문서가 아니라,
> 오늘 내가 어떤 판단을 했고 어떤 개발 행동을 반복하거나 바꿔야 하는지 확인하기 위한 기록입니다.
> 각 항목은 길게 쓰기보다, 사실/판단/근거/다음 행동이 드러나게 구체적으로 작성하세요.

### 1. 오늘 완료한 것
- 오늘 실제로 끝낸 기능/수정/학습:
    - HTTP 구조에 대한 학습
    - 기능 구현 : HTTP Request 파서, HTTP 데이터를 담는 HttpRequest, HttpHeaders 클래스, 요청 디스패처, 정적 요청 처리 클래스
- (오늘의 목표 중에서) 아직 끝내지 못한 부분:
    - ResponseBody를 받아서 HttpResponse를 하는 유틸 클래스 구현
    - ResponseBody 클래스 구현
    - 잘게 커밋 쪼개기

### 2. 오늘 가장 의미있었던 기술적 판단
- 내가 내린 판단:
    - 요청 디스패처인 RequestDispatcher를 구현할 때, 경로 기반이 아닌 파일 확장자 기반으로 정적 요청/동적 요청을 구분하도록 로직을 설계
- 그렇게 판단한 이유:
    - 결국 정적 요청이란건 리소스를 요청하는 것이고, 동적 요청은 url 기반으로 요청을 하는 것이기에 정적 요청은 파일 확장자가 붙을 수 밖에 없다. 따라서 복잡하게 url을 기반으로 구분을 하는 것보다 파일확장자가 있냐없냐로 구분이 가능하다
- 고려했지만 선택하지 않은 대안:
    - url을 별도의 Set에 모아두는 방식도 생각을 하였다. 하지만 이 방식은 url이 수정이된다면 같이 수정이 일어나야 한다. 파일확장자 방식은 url 수정이 일어나도 코드 수정이 발생하지 않는다는 점을 고려해봤을 때 채택하지 않은 이 방식은 유지보수성에 좋지 않다
- 이 판단이 코드 품질/테스트/유지보수에 미친 영향:
    - 위의 내용 + `.` 을 기준으로 분리를 하고, 분리된 파일 확장자가 set에 존재하냐라는 간단한 로직으로 구현이 되어있기에 코드도 짧다

### 3. 가장 막혔던 문제와 해결 과정
- 문제 상황 또는 증상:
    - 설계 과정에서 RequestDispatcher가 핸들러만 찾아주는 역할을 할지, 아니면 핸들러를 직접 호출하고 ResonseBody를 return 해주도록 할지 고민하였다
- 고민 과정:
    - 전자의 경우에는 요청을 라우팅만 하면 되기에 핸들러를 찾아주기만 하는 역할 분리가 명확하다. 하지만 별도로 요청이 추가될 때마다 핸들러 목록을 업데이트 해줘야하는 약간의 불편함이 존재한다
    - 후자의 경우에는 직접 호출하고, 결과를 반환해야 한다는 추가적인 역할이 부여된다. 책임 분리가 따라서 불명확해진다.
- 해결 방법:
    - 전자의 방식으로 구현을 하고 있다. 결국 요청을 처리하는 것은 그 과정에서 공통 부가작업(예를 들어, 요청을 하기 전에 인가가 필요한 요청인지, 자동으로 로그를 기록하게 할 것인지 등)이 있을 수 있다. 이를 디스패처가 담당하는 것은 책임이 과한 것이라 판단하였고, 핸들러를 찾는 클래스와 핸들러를 기반으로 요청을 처리하는 클래스를 따로 설계하도록 하였다
- 다음에 비슷한 문제가 생기면 먼저 확인할 것:
    - 처음에는 좀 더 구현이 쉬울 거 같은 후자의 방법이 끌렸지만, 요청을 처리하고 결과를 받는 과정에서 단순 리턴만 하는 게 아닌 내가 생각하지 못한 부가적인 공통 작업이 들어갈 수 있음을 깨달았다. 이를 통해 다음번에는 역할을 분리할 때 내가 생각치못한 공통로직이나 부가적인 작업이 있는지 확인을 해서 책임을 분리해야겠다

### 4. 오늘 잘한 행동(개발과 관련된 것)
- 어떤 행동이었는지:
    - 설계를 내 힘으로 스스로 해본 것
- 그 접근이 도움이 된 이유:
    - 이전에는 gpt에게 "~~이런 요구사항을 구현하려면 어떻게 해야돼?" 라는 식으로 설계를 나혼자 하지 않고 gpt에게 맡기고 있었다. 왜냐하면 그러한 경험이 부족했기 때문이다.
    - 하지만 이번 소프티어에서는 코드작성에서는 도움을 받을지언정 설계를 하는 과정에서는 내가 스스로 해보는 연습을 하고자 한다. 직접 하고나니 나만의 논리적인 구조를 찾아가는 과정도 재밌었고, 이 과정에서 객체지향적인 코드, 유지보수가 편하고 확장성/재사용성이 좋은 구조 및 코드를 고민할 수 있었다.
- 다음에도 반복하고 싶은 점:
    - 설계는 나 혼자하되, 추가적으로 더 좋게 설계할 수 있는 부분에 대해서는 AI의 도움을 받아서 힌트를 얻고 설계를 해봐야겠다

### 5. 오늘 아쉬웠던 행동(개발과 관련된 것)
- 비효율적이었던 접근/놓치거나 부족했던 점:
    - Http 요청에 대한 출력을 할 때 쓰는 OutputStream에 대해서 자세히 알아보지 않고 HttpResonse 출력 유틸 클래스를 작성 시 AI에게 의존하려고 했던 행위
- 그렇게 된 원인:
    - 출력을 하는 것은 그리 중요한 로직이 아니라고 판단을 내렸고, 이는 AI의 도움을 받고자 판단을 내렸다. 하지만 결국 다양한 응답 타입에 대한 HttpResponse를 지원하려면 출력 클래스에 대해서도 잘 알아야 했다
- 다음에는 다르게 해볼 구체적인 행동:
    - 덜 중요한 부분은 빠르게 AI의 도움을 받되, 설계적인 측면에서 필요한 개념의 경우 제대로 이해를 하고 로직을 구성해야겠다

### 1. 내가 최종 선택한 가장 잘 작성했다고 생각하는 프롬프트
원본 프롬프트:
DataOutputStream 이 왜 http 서버에서는 적합하지 않은거지? 왜냐하면 html, css 파일과 같이 정적 리소스 파일을 요청하는 것도 출력은 바이너리 파일로 해야되지 않나? json 으로 출력해야 되는 경우에만 문자열 출력을 고민해야 할건데 이럴빠엔 바이너리 출력을 택하면 되는거 아닌가?
선택한 이유:
질문 + 내가 이해하고 있는 내용(그렇게 생각한 이유) + 예외 케이스를 고려한 보충 설명 구조로 되어있어서 내가 잘못알고 있는 부분이 무엇인지, 올바른 정보는 무엇인지 지적을 받음으로써 어느 부분을 잘못알고 있는지 메타인지를 할 수 있었다

### 2. 원본 프롬프트 구조 분석
- 문제 상황: DataOutputStream이 HTTP 서버에 적합하지 않다는 이유를 이해하지 못하고 있다.
- 원하는 결과를 어떻게 요구했는가: 본인이 이해하고 있는 점을 근거로 DataOutputStream이 HTTP 서버에 적합하지 않은 이유를 설명해 달라고 요청했다.
- 필요한 맥락이 포함되어 있는가: HTTP 서버 구현 중이라는 것을 해당 프롬프트에는 없지만 이전 프롬프트에 작성해서 전달하였다.정적 리소스도 바이너리로 전송해야되지 얂냐는 배경을 함께 제시했다.
- 판단 기준이나 제약 조건이 있는가: 없다
- 출력 방식이 명확한가: DataOutputStream과 OutputStream의 차이를 비교·설명하는 형태를 기대지만 이에 대해서는 별도로 지정하지 않았다.
- 나의 가설, 우려, 의도가 드러나는가: "어차피 모두 byte로 전송되는데 DataOutputStream을 써도 되는 것 아닌가?"라는 가설과 의문이 드러난다.

### 3. AI에게 프롬프트를 개선 요청하기
- 프롬프트 개선안
```
HTTP 서버를 구현하면서 DataOutputStream과 OutputStream의 차이를 이해하려고 합니다.

일반적으로 HTTP 응답에서는 DataOutputStream보다 OutputStream을 사용하는 것이 적합하다고 하는데, 그 이유가 잘 이해되지 않습니다.

제 생각에는 HTML, CSS, 이미지와 같은 정적 리소스도 최종적으로는 모두 byte 형태로 전송되고, JSON 역시 문자열을 byte로 변환해서 전송합니다. 그렇다면 결국 HTTP 응답은 모두 byte를 전송하는 것인데, 왜 DataOutputStream을 사용하는 것이 적합하지 않다고 하는 걸까요?

단순히 "HTTP는 텍스트 프로토콜이기 때문"이라는 설명이 아니라,

OutputStream과 DataOutputStream이 내부적으로 어떤 차이가 있는지,
HTTP 프로토콜과 각각이 어떻게 맞물리는지,
정적 리소스(HTML, CSS, 이미지)와 JSON 응답을 예시로 들어 비교하면서 설명해 주세요.

또한 **"모든 응답이 결국 byte로 전송되는데도 OutputStream을 사용하는 이유"**를 중심으로 설명해 주세요.
```

### 4. AI가 개선한 프롬프트 분석
- 맥락
    - 어떤 상황인지, 그리고 무엇을 요구하는 지에 대해서 목적을 제시하였다
    - 내가 가지고 있는 지식을 기반으로 내가 가지고 있는 가설을 명확히 드러내었다
- 목표
    - 단순한 설명을 요구하는 게 아니라, 구체적으로 어떤 것들에 대한 설명을 요구하는지 목표를 구체적으로 제시했다
- 제약 조건
    - "HTTP는 텍스트 프로토콜이라서"와 같은 단편적인 설명이 아닌, 내부 동작과 예시를 포함한 논리적인 설명을 요구했다.
    - 다른 정적 리소스도 예시를 들어 설명해달라고 요청하였다.
- 추가된 출력 형식
    - 내부 동작, HTTP 와의 관계, 예시 비교, 결론의 흐름으로 답변하도록 제시하였다
- 답변이 좋아질 것이라 예측하는 이유
    - 종합적으로 보았을 때 질문의 배경과 사용자의 가설을 기반으로 사용자가 잘못 알고 있는 부분에 대해서 정확히 짚어주고 정해진 규격대로 설명을 하면서 의문을 해결할 수 있는 좋은 답변을 내놓을 것이라 예측된다
    - 설명에 대한 범위와 비교군을 요청하였기에 개념간 차이를 잘 이해할 수 있을 것이다
    - 맥락을 제공하고 구체적인 목표로 인해 추상적인 답변이 아닌 내부동작 및 원리 중심의 답변과 HTTP 서버를 구현하면서 응답에 대해 추가적으로 고려해야 되는 부분들도 기대한다
